中国移动经营分析系统数据质量管理办法

 

（试行）

 

1       概述

1.1     背景

中国移动经营分析系统自建设运营以来，为中国移动的经营决策、市场运营管理和营销服务提供了重要的系统支撑。但随着业务的快速增长和日趋复杂，经营分析系统中的数据流转和处理环节越来越多，数据管理越来越复杂，数据质量保障的难度也越来越大。

为进一步规范、强化数据质量管理工作，有限公司特制定《中国移动经营分析系统数据质量管理办法》(以下简称“《办法》”)，为总部及各省公司经营分析系统的数据质量管理工作提供系统、合理、有效的规范和指导。

1.2     目标

经营分析系统的数据源由原来单一的业务支撑系统、MIS系统逐步扩充到网管系统、数据业务平台、集团客户业务平台、第三方数据（如市场调查、其它咨询公司）等，日趋复杂的数据逻辑对经营分析系统数据质量提出越来越高的要求。数据质量管理是一个持续发展、不断完善的过程，立足于实现如下目标：

1、建立相应的管理机制，明确职责，从管理的角度加强和完善对经营分析系统数据质量的控制；

2、在需求开发过程中，与源系统及需求部门建立协同管理机制，实现设计、开发、测试及上线的流程化管理，并建立业务需求的接入标准化、数据模型的标准化、统计口径的标准化以及指标体系的标准化；

3、在日常运维监控中，借助元数据、监控、工作流等手段建立数据处理的全流程保障机制，在各个环节保证数据的准确性，一致性，完整性和可靠性；

4、定期对数据质量进行评估，形成问题通报机制，并对历史出现的问题进行总结，形成知识积累，持续促进对数据质量的管控；

5、通过技术和管理两个方面的持续提升，最终建立企业级数据的生产、加工和使用的全流程质量管控，让各级公司、各相关部门信任、依赖经营分析系统，使企业在激烈的市场竞争环境中获得更加精确的决策数据信息支撑。

1.3     内容简介

本管理办法针对目前中国移动经营分析系统数据质量的实际情况，分析了影响数据质量的诸多因素，明确了相关系统和部门的角色和日常职责，规定了质量控制相关环节的管理机制、管理办法和管理流程。

各章节的简要说明如下：

第1章－概述：描述《办法》的背景、目标、内容简介、适用范围、起草单位、解释权和参考文献等概要信息；

第2章－数据质量影响因素：从需求过程、数据源、指标口径、系统运营本身等几个方面详细描述了目前经营分析系统常见数据质量问题以及产生原因；

第3章－数据质量管理机制：描述为了保证数据质量管理工作有序进行而建立的数据质量管理团队组织机构，定义了数据质量管理团队成员的角色、职责，并从事前、事中、事后三个阶段定义了数据质量管理工作中的相关机制和制度；

第4章－数据质量管理流程：承接并依据上一章三个阶段的数据质量管理机制，从流程角度对三个阶段的数据质量控制予以阐述，明确了常见数据质量问题的处理流程；

第5章－省公司对总部经分系统的数据保障：介绍了总部一级经营分析系统有关数据质量管理方面的方法和内容，包括管理和流程的检查以及对数据的检查，用于指导省公司在保障一级经营分析系统数据质量方面的工作。

1.4     适用范围

本办法适用于中国移动总部及各省（自治区、直辖市）公司经营分析系统。

办法的增补、修订及解释权属中国移动通信有限公司。在此之前的文件如与本办法产生矛盾，均按本办法执行。

2       数据质量影响因素

自经营分析系统建设运营以来，中国移动初步建立并不断完善两级经营分析系统的数据质量管理体系，实现了基本的数据质量监控和问题处理流程。但随着公司各项业务的快速发展，以及数据质量管理工作的不断深入，出现了一些新的情况和问题，对公司的业务发展以及决策造成了困扰。

当前面临的数据质量问题，按照来源和具体原因，主要分为以下几类：

n        需求过程引发的问题；

n        数据源引发的问题；

n        统计口径引发的问题；

n        经分系统自身的问题。

2.1     需求过程引发的问题

需求过程引发的问题主要指需求设计、开发、测试、上线等过程中引发的数据质量问题。

此类问题的原因主要是因为需求过程中的管理机制和流程不健全导致：

1.        从管理流程上看，部分需求的审批、开发及上线管理流程较为简单，缺乏有效的需求审核和规划；缺乏全流程的开发质量控制；缺乏明确的元数据定义与跟踪、需求影响分析等；

2.        从需求内容上看，相关部门提出的需求通常采用工单或邮件等方式流转，需求大多不明确或口径定义不清晰，在沟通过程中造成需求信息的丢失和准确性的降低，所以需求提出人员应尽可能明确和细化需求的内容，并与业务支撑部门的分析、开发人员沟通形成精确且标准化的业务口径，用以指导开发，保障开发出的结果与需求的完全一致；

3.        从组织人员上看，需求过程中涉及相关部门的需求提出人员、需求使用人员、业务支撑部门的分析人员、开发人员、测试人员及运维人员等，但实际各个环节的人员配置不完善、或职责分工不明确，造成需求无法有效沟通或责任推诿现象；由于各开发环节人员对业务的理解以及所处的角度和关注点不同，经常造成相关信息口径、库表定义及与现有系统的融合方面出现不一致或冲突，出现质量问题。

上述影响因素表明，相关业务部门在需求过程中的作用非常重要，如果需求部门对需求描述不清晰或缺乏审核和规划，业务支撑部门开发出的质量自然难以得到保障，数据质量问题就无法得到彻底的根治和预防；而需求上线后，需要需求使用部门和业务支撑部门共同保障，出现问题时共同配合解决。所以保障数据质量并不是业务支撑一个部门的责任，而是相关部门共同的职责。

2.2     数据源引发的问题

数据源引发的问题是指由于上游数据源不完善或不规范等原因，导致作为下游的经营分析系统受到影响，出现数据质量问题。

数据源引发的数据质量问题是经营分析系统数据质量问题的主要来源，主要原因有以下几方面：

1.        信息不正确：源系统提供的数据无效或错误，违反数据约束规则、业务规则等；

2.        信息不完整：源系统中某些信息缺失或未填充，虽然不影响源系统正常运转，但这些信息的缺失会严重影响数据仓库的应用分析，如客户职业分类信息、入网渠道信息以及GPRS话单IMEI信息缺失等；

3.        信息不一致：当同一信息内容来自多个源系统时，存在冲突和差异，或者同一业务系统内部的冗余信息之间存在冲突，或当同一源系统的部分数据发生了变更，而未及时通知经营分析系统，造成经营分析系统与其有相关性的其它数据准确性出现问题。

此外，源系统数据字典的完整性维护也可能引发数据质量问题，当数据源可能新增或者变更了数据字典，容易造成经营分析系统解析错误。

为此，需要建立源系统与经分系统的数据质量协同管理机制，规范源数据的接入和管理，而且，由于数据源造成的数据质量问题占比非常高，需要经分系统配备专人来负责保障源系统的接口和数据质量。

2.3     统计口径引发问题

统计口径引发的问题主要指KPI、报表等指标口径中存在的数据质量问题，包括指标准确性、一致性及完整性问题等。

此类问题产生的原因主要有以下三类：

1.        不同源系统或不同业务部门，对相同名称的指标定义和口径不同，造成最终的指标统计结果出现差异；

2.        业务部门在描述或定义指标口径时，存在许多不明确和不完善的地方,造成了指标口径的二义性；

3.        业务人员、开发人员等对需求中的指标口径的理解和沟通存在偏差，造成最终开发的统计口径与实际的业务口径不符，出现问题。

2.4     经分系统自身的问题

经分系统自身的问题主要是指经营分析系统在开发建设、日常运营和维护过程中引发的数据质量问题。

经营分析系统经过多期的建设，虽不断走向成熟，但系统本身仍然存在着一些问题，比如数据模型质量问题、系统升级数据信息丢失问题、ETL数据清洗不够彻底带来的问题以及数据仓库流程调度等问题，原因主要有：

1.        数据仓库的建设过程中，由于缺乏规范化、体系化的建设思路，造成系统架构、数据模型、处理流程等不够优化和合理，从而带来数据质量问题；

2.        在经营分析系统运营中，由于普遍缺乏完善的日常管理和维护流程，在处理数据或监控时，没有可以依据和规范的处理手段，导致操作出现失误或遗漏，造成数据质量问题。

针对上述影响数据质量的各个因素，需要建立健全管理制度和流程，保证每个影响因素都能的得到解决和有效的控制，从而全面保障和提升经营分析系统的数据质量。

3       数据质量管理机制

数据质量管理机制包含数据质量保障团队的组织架构、职责要求、相关管理制度和方法，是对经营分析系统数据质量管理各项工作的整体描述。建立完善的数据质量管理机制的目的如下：

1.        确保经分系统的数据能够真实、准确、完整地反映移动业务发展情况，为经分系统正常运行、稳步提升数据质量提供有力保障；

2.        实现业务与技术的有效衔接，规范业务开展行为，明确经分系统业务需求实现过程中的难点、重点以及数据质量管理工作的薄弱环节，为解决影响经分数据质量的障碍提供一套行之有效的解决办法；

3.        为数据质量的考核和问题解决责任方的确定提供依据，公正、公平、客观地评价经分数据质量问题，并进行职责追究。

3.1     组织架构

3.1.1    组织结构图

数据质量管理组织结构图如下：

 

    数据质量管理团队由13个角色组成，包括领导小组组长（公司主管副总）、领导小组副组长（各部门领导）、执行小组组长（项目负责人）、业务部门数据质量管理员、经分系统数据质量管理员、源系统数据质量管理员（由对口部门或项目分别指派）、需求质量管控组、数据质量监控组、质量问题处理组、申告处理组、系统支撑组、专家组、数据协同处理组。

3.1.2    组织机构职责

n        制定数据质量检错标准；

n        制定数据质量管理、保障和维护流程；

n        建立数据质量管控机制，协调各组织机构的沟通和工作配合，如与数据源系统维护机构、经营分析系统管理人员协调解决数据源系统接口问题；

n        监控数据质量问题处理的整个流程并进行结果评估；

n        负责日常监控、处理数据质量问题和发布处理结果，提供问题咨询服务；

n        定期（如每月、每季）或不定期进行数据质量状况评估；

n        负责处理数据质量需求变更的评估与实施。

3.1.3    角色与职责

3.1.3.1  领导小组组长

n        人员要求：应由主管经营分析系统的公司副总担任。

n        工作职责：为经营分析系统的数据质量管理工作提供宏观方向指导，并予以授权。

3.1.3.2  领导小组成员

n        人员要求：应由主管经营分析系统的省公司副总、业务部门副总、业务支撑部门副总担任。

n        工作职责：为省级业务系统、经营分析系统的数据质量管理工作提供宏观方向指导，并予以授权。

3.1.3.3  项目负责人

n        人员要求：应由经营分析系统建设和维护单位的分管经理担任。

n        工作职责：主要包括：

a)        全面负责经营分析系统的建设和维护；

b)        为经营分析系统数据质量管理工作提供方向指引，并对具体工作落实提供指导意见；

c)        负责数据质量管理团队工作相关的组织机构或系统间协调。

3.1.3.4  业务部门数据质量管理员

n        人员要求：业务部门数据质量管理员是业务系统数据质量管理工作的主要角色，是数据质量管理团队的工作管理核心。应由市场、数据、集客、财务、网络、客服等相关部门指派本部门分析人员担任。

n        工作职责：主要包括：

a)        制定业务系统的数据操作规范和业务指标定义标准；

b)        监督并管理相应业务系统的业务开展和系统操作活动是否遵从数据操作规范；

c)        负责管理与协调需求质量管控组日常工作，组织解决业务需求过程中引发的数据质量问题；

d)       负责经营分析系统与业务系统间的需求确认协调和改进工作落实；

e)        参与质量方案的制定和评估。

3.1.3.5  经分系统数据质量管理员

n        人员要求：经分系统数据质量管理员是经分系统数据质量管理中最主要的角色，是数据质量管理工作的管理核心，负责管理与协调数据质量监控组、质量问题处理组、申告处理组、系统支撑组等日常工作。一般由经分系统项目负责人指派专员担任。

n        工作职责：主要包括：

a)        全面主持经分系统数据质量管理各项工作的开展和落实，并对经分系统数据质量工作的开展提出有效建议；协调相关的组织机构或源系统，组织对异常或有争议数据质量问题进行协商解决；

b)        制定经营分析系统数据质量管理制度、管理规范、管理流程、考核办法、及相关文档模板等；

c)        组织建设和维护数据质量管理子系统、数据质量知识库，对数据质量管理子系统和知识库的建设提出需求与建议，以技术手段为辅助提高数据质量问题发现和沟通处理效率；

d)       定期（每月）或不定期地组织进行数据质量评估（包括业务需求影响评估、源系统数据影响评估等），提交评质量问题评估报告，可作为质量问题通报的依据；

e)        组织数据质量相关知识及流程的培训；

f)         组织编写和维护数据质量管理工作相关日常管理文档，如数据质量检查报告。

3.1.3.6  源系统数据质量管理员

n        人员要求：源系统数据质量管理员是源系统接口数据质量管理工作的核心，主要负责管理和协调数据协同处理组的各项工作，协调解决各类源系统数据问题的。一般由负责源系统建设和维护的部门指派人员担任。

n        工作职责，主要包括：

a)        负责管理与监督相关源系统数据质量工作的的开展和落实，协调并落实经营分析系统与业务系统间的问题解决；

b)        对数据协同处理组的工作开展提出方向指导和建议；

c)        制定和优化源系统数据接口的质量管理保障规范、接口开发流程、接口数据质量考核办法、及相关文档模板等；

d)       负责管理和组织源系统、经分系统的数据接口开发、维护、考核工作；

e)        组织编写源系统接口开发和维护的相关日常管理文档，并对相关文档进行维护。

f)         参与数据质量各种文档和流程的评审。

3.1.3.7  需求质量管控组

n        人员要求：需求质量管控组，应由业务部门和经分系统分别指派专员组成，在业务部门数据质量管理员和经分系统数据质量管理员组的引导下开展工作，其主要职能是对新业务需求的提出、定义、开发和验证过程中保障需求和源系统数据的准确性。业务部门专员，可以由固定的业务部门数据质量管理员及需求提出人组成。业务部门数据质量管理员负责相关业务需求的统一掌控，进行需求规划、进行支撑可行性、数据质量风险的评估等；需求提出人负责在整个需求提出、实现、上线运行整阶段的数据质量跟踪工作。

n        工作职责：主要包括：

a)        遵从需求质量管控流程，需要明确需求目的、需求内容，尽可能地避免统计口径不明确、指标定义二义性、需求方与接收方沟通不充分等问题，形成详细、明确、具备指导意义的需求规格说明书；

b)        跟踪需求过程，协调解决需求过程中存在的数据质量问题；

c)        监督需求上线情况，保障和解决新需求上线期间（如3个月）的数据质量问题直至需求稳定；

d)       参与数据质量各种文档和流程的评审。

3.1.3.8  数据质量监控组

n        人员要求：应由经分系统指派专员组成。

n        工作职责：主要包括：

a)        接受经分系统数据质量管理员的任务安排和监督，及时报告工作进展；

3.2     管理机制

根据经营分析系统数据质量管理要求，将数据质量管理机制落实并细分到经分系统各个环节中：

n        事先预防：在需求开发阶段进行事前预防，建立数据质量协同管理机制和数据标准化管理机制；

n        事中监控：在系统运维过程中，做好事中监控，建立数据质量监控预警管理机制、数据质量问题处理管理机制、信息发布管理机制和申告管理机制；

n        事后总结：事后需要及时进行总结提升，建立质量问题通报机制、考核管理机制和数据质量知识管理机制。

3.2.1    数据质量协同管理

从经分系统数据质量影响因素来看，数据质量管理不是单系统、单部门的问题，实现全面数据质量管理需要跨系统、跨部门的协作。

数据质量协同管理涉及三个层面：

1.        首先是与数据源系统进行协同管理，共同保障上下游之间数据的一致性和准确性；

2.        其次是与需求部门进行协同管理，定义统一的业务需求管理办法，明确需求的业务逻辑和统计口径，保证在需求接入过程中对需求有清晰的描述，在验收和使用过程中共同保障需求结果的准确性；

3.        最后是总部与省公司之间的协同管理，明确总部与省公司之间的责任和义务，通过实施相应的技术和管理措施，确保全网数据的一致性和准确性。

3.2.1.1  数据源协同管理机制

数据质量问题大部分是由源系统引起，保证与源系统有效的沟通是减少数据质量问题的关键。数据源协同管理应包括以下几个方面：

1.        由数据源系统变更引起的系统割接，要求数据源系统能够及时通知经分系统。经分系统同步反馈割接结果，该结果用来作为数据源系统割接成功与否的判断标准。

2.        数据源系统涉及的业务需求，数据源系统通知经分系统，经分系统必须参加数据源系统业务需求的评审，对影响经分业务的需求，经分系统根据评审结果进行改造。

3.        数据源参数配置变更，数据源系统必须及时的通知经分系统，由经分系统根据变更内容进行相应的变更操作，并发布变更结果。

4.        由经分系统发现并且是因为数据源问题引起的数据质量问题，经分系统需要回传给数据源系统改进，避免相同的问题重复发生。

5.        对于数据源割接、数据源参数变更、经分发现的数据源问题引起的问题，在数据源系统完成改造之后，经分系统必须与数据源系统进行严格的数据校验，保证问题解决后的数据同步。

3.2.1.2  部门间协同管理机制

部门间协同管理包括需求提出涉及的各业务部门，需求开发数据接入涉及的各源系统、需求上线过程中经分内部各开发部门，以及需求验收使用的各部门。通过部门间的协同管理让需求开发过程有序的进行。

需求提交人在提交需求时，必须通过经分的需求提交流程上提交，否则不予受理，需求提交时需首先明确需求目的、涉及业务的指标口径、需求的业务类型，并对其需求所产资源的有效刷新时间、资源查阅权限、要求的上线展示位置给予确认。需求审批部门应对需求提出部门提交的内容给予审核，对不符合要求的需求退回需求提出部门，审批通过的需求提交需求开发部门进行开发。一旦进入需求开发阶段需求的内容将不能进行修改。

源系统接口协议管理包括制定数据源接口的传输规范、文件内容的传输标准化、保证在数据源接口在数据上源系统与经分系统理解的一致性。

业务人员在提出需求的过程，需要严格按照需求开发质量管控流程进行需求的提出，到后期的需求协助跟进，加强与数据质量专员的沟通，保证数据质量问题尽量减少。

在需求开发过程中，开发人员、测试人员以及后期的质量运维人员都需要按照需求开发过程规范进行相关的工作。减少在需求开发过程中的人为错误、以及反复开发的过程。

需求完成上线之后，需求提出部门、需求使用人员、需求考核负责人员必须针对需求的上线应用的效果，对需求进行验收，给出需求上线评价，让需求更加有效的应用。

3.2.1.3  总部、省公司两级协同管理机制

总部与省公司之间既是两级系统的协同，又是两级机构的协同，是数据质量协同管理中至关重要的组成部分。总部、省公司两家协同管理明确总部与省公司之间的责任和义务，通过实施相应的技术和管理措施，确保全网数据的一致性与准确性。

比如，在日常工作中，总部与省公司可以通过元数据互通方式实现基础数据结构与数据转换流程的协同，全面把握数据的来龙去脉；通过预警上报的方式提前告知即将实施的割接及影响，可能的业务波动等等。办法第五章具体讲述省公司对总部经分的接口数据保障要求。

3.2.2    数据标准化管理

经分系统由数据仓库、数据集市、数据挖掘等多个系统组合，各系统的定位不同，应用百花齐放，逻辑模型也存在较大的差异，同一份的原始数据，在不同的系统中格式各不相同，不能复用。数据仓库、数据集市、数据挖掘等经分系统各子系统的数据来源都是数据数据仓库，所以在数据源层，各系统是统一的。因此，需要建立一个核心基础模型，在各系统中保持一致。

数据质量的标准化主要包括数据模型标准化、指标体系标准化等。通过标准化数据模型，形成统一的数据结构和数据定义，避免同一指标由于数据模型差异导致数据不一致；通过标准化指标体系，能够保证各数据口径一致，让指标口径对于业务人员、开发人员来说都是“可读”的，不会出现同一指标用不同的统计口径，同一个统计口径有两种不同的“叫法”。

3.2.2.1  数据模型标准化

模型的标准化主要包含逻辑模型以及物理模型两部分的内容。

1.        逻辑模型标准化

数据模型的标准化首先必须从逻辑模型上进行数据建模，确认核心模型，然后在核心模型的基础上进行根据业务扩展。

核心逻辑基础模型的特性：

n        口径一致，核心基础模型内的命名统一，不存在命名相同，口径不一致的情况。

n        数据无冗余，相同的指标不会在不同的表中出现。

n        各系统的模型结构、数据是一致的，是整个经分模型的核心。

n        核心基础模型由多个系统的核心模型共享组成，各系统间相互同步，如数据仓库的三户模型、数据集市的县公司归属、数据挖掘的客户统一视图等。

 

从组成上来说，标准逻辑模型分基础数据逻辑层、汇总数据逻辑层、信息子层，三个层次。

基础数据逻辑层，因为受业务系统数据模型的限制，所以采用以本省核心基础模型为基础，集团标准逻辑模型做参考的方式确定。

汇总数据逻辑层、信息子层，采用以集团标准逻辑模型为基础，本省核心基础模型做补充的方式确定。

2.        物理模型标准化

物理模型的标准化主要遵从集团模型为参考，设计结构清晰，分主题存储，可扩展的，稳定性高的，口径统一的单一视图的数据仓库模型。物理模型主要的设计方法有：

n        主题域清晰：严格按照集团主题域的概念对模型进行划分，然后通过对业务含义的分析将实体和主题域进行一一映射，确保模型整体结构清晰。

n        对象集中存储：对模型中同类别的实体，比如SP信息，全部归纳到参与方主题域，并对来自不同平台的SP信息进行整合处理，确保企业范围内的SP信息是一致的。

n        规整参照关系：经过对数据源和业务的理解，重新规划实体之间的关系，确保关系能够正确体现业务规则和参照完整性，对数据质量有很大的帮助。

n        命名规范化：统一的命名方式，相同的指标使用统一的命名方式。尽量采用业界通用的名称。

n        统一口径：对业务代码进行统一整理并进行统一编码处理，这样在应用中可以统一计算口径，建立企业范围内的数据标准。

n        模型抽象化：业务分离与数据源分离，抽取共有信息，将模型进行抽象，实现数据与应用的解耦，提高模型的复用度，减少数据冗余，确保模型的高扩展性和高稳定性。

3.2.2.2  指标体系标准化

数据产生于企业的各个生产环节，存放在各类生产系统和分析系统中，用于反映企业的运营状况。指标是描述同类现象总体数量特征，是获取数据的依据。指标体系就是由一系列指标组成的，用来描述企业各个方面相互依存、相互制约关系的指标集合。在数据质量管理整个过程中，业务部分主要关心的就是指标，整个系统的数据权威性也是通过指标展现出来。

1.        指标体系化管理主要实现以下目标：

n        建立科学、合理和稳定的指标架构体系，保障数据字典稳定、健康；

n        基于指标架构体系，对集中数据分析系统指标进行梳理，对指标的定义、口径进行规范、统一，保障关键业务指标口径一致；

n        建立数据字典支撑体系，使数据字典落地，促进数据字典价值发挥；

n        建立数据字典管理体系，以使数据字典可持续健康发展。

2.        指标体系建设要求：

n        要求提供指标信息定义服务，提供给业务应用调用，业务人员方便快捷了解指标的含义信息；

n        数据字典提供检索和导航功能，业务人员使用本功能快速获取所需的指标定义及在系统中具体的指标数据信息；

n        数据字典信息提供直观的展现功能，可以从实体对象、生产流程、业务视角等不同视角展现指标信息。

n        对于指标及维度，系统应建立编码规则进行分类管理及分层管理、维护，并按各种常用分析主题定制约束条件模板；同时系统应支持通过模糊查询功能快速定位；

n        系统应可定义指标及维度的生命周期，并记录指标及维度的使用频度、提出单位等信息，同时自动根据指标及维度的生命周期将失效指标及维度归档。

n        系统提供指标与报表之间、指标与指标之间对应关系（依赖关系）、指标与源数据（数据表及字段）关系的管理功能，支持从指标查报表、从报表查指标等的双向查询功能、支持报表、指标、源数据（数据表及字段）间的逻辑关系管理，并在界面灵活展现。指标及维度涉及的中间层或底层数据应能按地域等维度进行重处理。

n        建立完整、统一的指标管理模板，支持指标信息输入及维护；

3.        指标上线过程管理：

为了保障经分系统数据的标准性和规范性，特针对经分系统标准指标（标准指标库和指标快讯）制定管理维护规则，以保障经分数据查阅的便捷性和可信度。

n        标准指标的建立

标准指标的新增和调整由指标所属业务部门和市场经营部相应主管提出需求申请，由业务支撑部门开发实现，申请必需注明指标的业务注释口径和时限要求。

n        标准指标的开发

业务支撑部门在进行标准指标的开发和调整时，一定要按照标准指标库的开发规范进行，指标快讯显示的指标一定要先在标准指标库开发实现，才可在指标快讯区配置展示。

n        标准指标的上线展示

标准指标开发完毕后，要首先由业务支撑部门经分系统相应技术支撑主管审核，再提交给市场经营部业务支撑主管审核，审核通过后方可上线展示。

n        标准指标的维护

针对上线展示的标准指标，每月由市场经营部相应业务支撑主管进行查阅和梳理，针对数据的有效性、是否刷新等情况进行检查，发现超出有效期的数据要尽快安排下线处理；

3.2.3    数据质量监控预警

数据质量监控预警管理机制是根据预先配置的规则或算法，进行数据采集和规则验证，并依据稽核和检查过程中发现的数据质量异常情况进行告警。数据质量监控环节包括（但不限于）如下几个环节：

n        监控预警规则配置

n        监控数据采集

n        数据质量监控

Ø        接口数据稽核

Ø        处理过程检查

Ø        指标监控和告警管理

3.2.3.1  监控预警规则配置

数据质量监控以被监控对象为中心，从数据采集、检查、到告警等每个过程都需要进行规则配置。

在数据质量管理工作特别是监控与告警环节中，质量管理员经验的积累与共享越来越被重视，为了使这些经验得到积累和有效沉淀，规则库建设的概念也就应运而生。规则库的建设，是一个将质量管理员的经验进行总结沉淀的过程，应用到数据质量管理系统监控功能的建设中，可以有效减少数据质量监控人员的日常维护工作量、提高发现问题和响应处理的效率。

规则类别，包括但不限于：采集规则、检查规则、告警规则、告警接收配置。应尽可能包含所有数据采集到的范围，如接口处理的检查规则、接口关键业务指标的检查规则、ETL处理过程的检查规则、仓库处理过程的检查规则、业务指标数据的检查规则。

n        采集规则：主要配置被采集对象和算法，包括采集规则名称、被采集对象所在的IP、端口、数据库及用户模式、采集对象、采集对象的属性和维度、采集算法及算法参数等。

n        检查和告警规则：主要配置被检查对象和算法，包括检查规则名称、规则类型、被检查对象、检查算法、参数项、常量（如阈值）、对应的异常告警规则、告警等级等。其中，业务指标数据的检查和告警规则，可分为指标波动检查规则、指标平衡性检查规则、指标关联性检查规则、指标加权波动检查规则等。

n        告警接收配置：主要配置告警接收人与检查规则的对应关系。如，接口处理信息、ETL过程告警、仓库处理告警，只传给数据质量监控和处理人员；而KPI指标检查告警，则可以传给数据质量监控人员、处理人员、数据质量管理员和相关部门接口人。其中主要配置接收人员、接收方式（如，短彩信、监控界面、邮件）、接收时间、接收内容及对应的内容格式等。

3.2.3.2  监控数据采集

根据数据质量监控内容的需要，在数据源系统和经营分析系统数据处理的相关环节配置采集点，采集数据质量监控信息。

1.        数据采集范围

包括但不限于：接口处理信息、接口关键业务指标数据、ETL处理过程、仓库处理过程、关键指标数据等。

2.        数据采集环节与角色职责

n        采集数据的需求收集：由数据质量管理员统一收集。

n        采集数据的配置：数据质量管理员统一收集后，由系统支撑人员定期变更和检查。

n        数据采集的验证：由数据质量监控人员进行验证，如有问题则反馈给数据质量管理员统一安排修改。

3.2.3.3  数据质量监控

数据质量监控的范围，包括接口数据稽核、处理过程检查、关键指标等。监控的周期，可以根据业务特性可以分为小时、日、月等。

1.        接口数据稽核

接口数据稽核是对源系统传到经营分析系统侧的源接口文件及数据、关键表、关键字段和关键指标进行数据稽核，验证数据的差异性及波动性。接口关键数据稽核的目的是尽早发现数据是否存在异常，及早做出应对，减少对经营分析系统的影响。

n        接口数据稽核内容：可分为，接口文件级及记录级检查、接口关键字段波动检查、源系统关键指标比对等。

n        工作环节与角色职责：接口数据稽核，一般由数据质量监控人员每日监控和处理。关键接口问题和告警等级高（后续影响大）的接口质量问题，将问题上升至数据质量管理员、数据源管理人员，由其协调处理。

2.        处理过程检查

处理过程检查是指根据预先配置的检查规则和质量检查度量，对数据处理过程的检查稽核。重点关注处理过程的及时性和准确性。

n        处理过程的检查内容：可分为ETL过程（清洗、转换、装载）监控和报错处理，仓库处理过程（如数据关键、汇总）的监控和报错处理。数据集市处理过程及应用展现过程等。

n        工作环节与角色职责：处理过程的监控和检查，一般由数据质量监控人员和处理每日进行日常维护。重点程序、重点实体、关键表关键字段的异常告警，需要根据问题原因的分类，分别上升至数据质量管理员、数据源管理人员，开发人员，由其协调处理。

3.        指标监控和告警管理

经营分析系统的指标种类很多（如用户数类指标、业务量类指标和收入类指标等），这些指标分布在经营分析系统的KPI、报表、上传和下发接口、挖掘分析等应用中。要及时、有效地发现关键指标的异动情况，需要系统支撑手段（指标检查规则配置和监控功能）作为辅助，以便及时、主动地将关键指标异动情况推送给监控人员和处理人员。

在所有数据处理环节和质量监控点中，指标是最终最直接为各部门和总部、省、市公司所感知的，因此，指标（特别是KPI、重点业务指标）的定义标准化规范化工作、监控日常化、处理流程化等管理工作，直接影响着使用人员对经营分析系统数据的依赖性和信任，也直接影响经营分析系统的推广和应用。因此，指标的监控、告警、处理、及持续优化的过程，将为最终实现“指标全流程管理”的目标奠定基础。

n        指标监控的检查内容：包括但不限于KPI指标、重点业务指标、上传和下发重点接口等。

n        工作环节与角色职责：指标异动情况（包括重点业务指标、上传或下发重点接口），一般由数据质量监控人员每日进行日常监控和原因分析，发布当日的数据质量报告（可以是短彩信、邮件、页面公告等形式）；每月KPI指标和涉及KPI指标的接口数据异动情况，由数据质量管理员进行跟进和分析，发布相应的数据质量报告（可以是短彩信、邮件、页面公告等形式）。

3.2.4    数据质量问题处理

数据质量问题处理，是将数据质量监控中和业务部门使用过程中发现的异常进行原因分析和定位，提交原因分析报告，并根据问题类型的定位、触发不同的问题处理子流程，直至最终解决。

数据质量问题处理的三个环节是问题发现、问题分析和处理、处理结果验证。

n        首先，“问题发现”的来源主要包括系统监控、使用人员申告和开发人员手工登记；

n        其次，“问题分析和处理”是根据问题类型的定位、经由项目负责人和数据质量管理员审批、分派到相关执行组处理的过程。

n        最后，“处理结果验证”是由问题发现人员验收并确认问题是否已修复。

 

质量问题的原因分类一般包括：

n        接口变更问题

n        数据源数据问题

n        系统故障问题

n        程序开发问题

n        口径理解问题

n        错误告警问题

 

根据各种问题原因分类执行不同的问题处理方案。

3.2.4.1  接口变更问题处理

接口变更问题，是指因经营分析系统业务需求引起的、或者数据源自身变更（网络调整、系统升级、系统割接、数据源接口变更等）的要求，经营分析系统需要进行相应的数据源变更处理。

接口变更问题的处理，遵循“数据源变更处理流程”，由发现人员进行问题登记，由数据源管理人员协调和处理。

3.2.4.2  数据源数据问题处理

数据源数据问题，是指数据源系统因系统或业务问题引起的、影响到提供给经营分析系统的接口数据不够准确，需要进行重传和重出数处理。

遵循“问题处理流程”和“数据重跑处理方案”，由发现人员进行问题登记，联系项目负责人，提出数据重跑请求和发起工单，由数据质量处理人员实施处理。

3.2.4.3  系统故障问题处理

系统故障问题，是指因经营分析系统故障（如数据库暂停、未建表）等引起的程序报错等，需要在系统故障修复后及时进行处理、和进行数据库维护。

系统故障问题的处理，遵循“问题处理流程”，由发现人员进行问题登记，由数据质量处理人员手工修复。

3.2.4.4  程序开发问题处理

程序开发问题，是指由于数据源数据变化或其它因素、导致数据库表的模型设计需要调整、或程序代码需要调整。

遵循“问题处理流程”，由发现人员进行问题登记，由系统开发人员进行修复。

3.2.4.5  口径理解问题

口径理解问题，是指由于开发人员和需求人员、各个接口负责人员之间信息不通畅，理解不一致，导致最后指标出现不符合逻辑，失真等情况的问题。

口径理解问题的处理，遵循“问题处理流程”，由发现人员进行问题登记，明确正确的取数口径或条件，由数据质量处理人员手工修复。

3.2.4.6  错误告警问题处理

错误告警问题，指因业务发展变化、需求变更或其他因素引起的监控规则设置不合理，需要进行规则变更。

错误告警问题的处理，遵循“规则变更流程”，由发现人员进行问题登记，由数据质量管理员实施规则变更。

3.2.5    信息发布管理

信息发布管理，指利用或改造现有的经营分析系统、邮件、短彩信网关等系统平台发布指定信息（如新需求上线、系统功能变更、数据异动原因通知等），是质量管理的一项基本工作。

经营分析系统主动进行信息发布，可以及时发布质量问题处理结果、质量问题原因和系统异动情况等，从而使数据质量问题的处理信息得到及时的交流和主动推送，从而加强经分系统与使用人员、管理人员的沟通。

1.        信息发布范围，主要包括：

n        经营分析系统新需求/功能上线（包括，新增应用，应用变更，功能改进，数据下发等）

n        数据质量异动情况（包括指标出数及时性情况、波动情况、原因说明、影响范围等）

n        源系统异动情况（包括源系统名称、情况说明、影响范围等）

n        系统割接情况（包括割接工程内容、工程实施时间、影响范围等）

n        其他需要进行通知下游系统和使用人员的信息

2.        信息发布形式和接收对象

n        经营分析系统前端公告（接收对象：经分系统使用人员）

n        经营分析系统报表异动公告（接收对象：经分系统使用人员）

n        短信 / 彩信 / 邮件（接收对象：项目负责人，数据质量管理员，数据质量监控组，及其他定制相关信息的人员）

3.        信息发布与角色职责

n        发布需求提出：经营分析系统项目所有人员均可提出。发布信息的请求由数据质量监控组进行接收和提交审核。

n        发布内容审批：由经分系统数据质量管理员统一审核，对于相关部门的个性化系统，可由业务部门数据质量管理员进行审核。

n        信息发布：由数据质量监控人员实施信息发布。

3.2.6    申告管理

申告管理，是指经营分析系统提供一个可供使用人员进行信息咨询和问题投诉的渠道，使用人员可以发起质量申告和问题登记，由经营分析系统项目的专职处理人员进行问题接收和分析，同时与申告人员进行充分沟通，并及时转入相关问题处理子流程、直至最终解决问题和回复申告人员。申告接收是数据质量问题发现的一种有效手段，做好申告处理是质量管理的一项基本工作，也是提升用户使用经营分析系统数据的一种必要手段。

1.      申告内容范围，包括但不限于：

n        口径咨询：包括指标定义咨询、业务开展与系统实现情况的咨询等。

n        数据疑问：如，数据未及时展现；数据明细与报表统计不一致；对指标数据存在疑问，要求核查并提供原因等。

n        其他：如账号权限问题，系统功能使用问题等。

2.      申告处理环节与角色职责

n        申告发起：由经营分析系统使用人员在申告管理平台发起（登记），必须填写的内容包括申告标题、申告对象在系统上的路径、申告内容和要求、相关申告的附件、申告发起人的联系方式等。也可直接拨打申告热线进行问题登记。

n        申告接收：由数据质量申告处理人员在平台上接收申告、并进行一对一地沟通和问题跟进。申告接收的响应要求在24小时以内（节假日除外）。

n        申告处理：指由数据质量申告处理人员接收申告后，对申告内容进行分析，判断申告的问题定位、根据问题类型的定位转入相应的处理流程，处理后需跟进反馈。对于申告方二次申告的情况，需重新处理并跟进反馈提出的申告。申告处理及时性的要求可根据实际系统支撑情况和问题处理复杂度进行调整，如数据源问题要求3个工作日内解决、非数据源问题要求2个工作日内解决等。

n        申告反馈：由数据质量申告处理人员向申告方反馈处理结果，申告方需反馈问题处理结果的满意度。

3.2.7    质量问题通报

为了定期掌握经分系统数据质量现状，明确数据质量问题产生的原因，界定相关责任，要建立数据质量问题通报制度，由业务支撑部门定期发起通报，将每时间段内收集的数据质量问题汇总分析，交由相关责任部门牵头在规定时间范围内解决对应的数据质量问题。

通报内容主要包括在该时间段内发现的数据质量问题情况（包括业务需求质量问题、源系统接口数据问题、经分系统开发质量问题等）、问题解决责任方、处理和改进要求、完成时间要求或问题反馈时间要求等。

3.2.8    考核管理

考核管理是一种监督机制，通过有效、合理的监督考核机制，引导经营分析系统各相关部门、各责任人更加有效、及时的解决问题，从而能够有效的提升经分系统的数据质量。数据质量考核主要包括对经分系统开发商的考核和对数据源系统的考核。

3.2.8.1  源系统考核管理

经分系统的数据都源于不同的外部系统，如BOSS系统、CRM系统、网管系统、数据业务平台等，为了保证经分的数据及时性、准确性，与源系统制定考核机制是必要的。

对源数据提供方为业务支撑部门内部所辖系统的，将及时性和准确性指标列入考核范围，由经分系统根据源数据的提供和异常处理情况进行考评。

对外部门所辖系统的考核管理，需借助公司内其主管部门对该系统开发商的考核给予实现。经分系统需每月定期形成外围系统的源数据提供和异常处理情况通报抄送给业务部门使用。对于一些较难解决的问题，长期没有很好的解决的，要协调上级部门加强考核制度，确保问题能够有效的处理。

考核指标

考核说明

接口到达及时性

数据源接口到达的及时性，按照接口传输规范，是否及时传输过来；每月进行数据源及时性进行发布通告。对于严重的问题进行扣分处理。

接口到达准确性

对数据源接口的数据准确性进行考核，保证数据的准确性。

如，通过同类接口故障出现次数进行考评，若3个月内同一接口同类故障（按故障现象划分故障类别）出现次数在3次及以上，则一般故障升级为严重故障；若3个月内同一接口同类故障出现次数在5次及以上，则严重故障升级为重大故障。

问题处理及时性

该指标主要考核源系统解决故障的及时性，避免发生故障之后问题不能及时的处理。

故障处理不区分节假日与工作日。重大故障发生后经分系统与源系统联系，并及时通报故障解决情况。

如，严重故障必须2个小时内解决问题，不然进行扣1分处理，如果超过3个小时没有解决进行扣2分处理，一天没有解决扣5分处理。

3.2.8.2  开发商考核管理

经分数据质量的考核涉及到团队较多，需要制定的考核范围较大，每个队伍的考核需要实际情况进行相应的考核规范制定。经分的数据质量考核主要包括经分系统提供给各部门使用的报表数据，如经分数据的出数及时性、准确性；对于经分系统参与运维的运维团队进行考核，如系统发生故障（主机宕机、磁带备份未成功、数据库故障）之后，对问题解决的情况进行考核；另外对于为经分提供的系统功能建设的IT部门进行考核，如需求提出之后，需求完成的质量，用户使用之后的满意度进行相应的考核。

 

考核指标

考核说明

统计及时率

经分系统应用需要的数据报表，能否及时的生成统计结果。

统计准确率

经分系统中提供统计数据的准确性。

需求及时率

由各部门向经分系统提出的需求，经分系统完成的需求的及时率。

需求完成满意度

经分系统在完成各部门提出的需求的过程中，需求提出人对需求完成情况的满意度。

如：非常满意考核绩效加1分；满意、一般则不用扣分，不满意则需要扣除绩效1分；非常不满意则扣除绩效2分。

运维考核

在经分系统运维支持过程中，需要对数据质量涉及的运维进行考核。

如系统故障发生之后的问题解决的及时性以及有效性进行考核。

3.2.9    数据质量知识管理

数据质量知识是在经营分析系统使用及运维过程中，由人工总结或者系统自动收集的有关数据及过程问题的处理经验总结。这些知识可作为今后数据质量问题解决方法的参考，并可以按关键字的形式进行索引和分类管理。

3.2.9.1   数据质量知识产生

数据质量知识主要来源于对数据质量问题处理的总结，同时，对数据质量问题的不同解决方案，以及对知识本身的评价，也是对数据质量知识的补充与完善。

数据质量知识中包括了以下经验的集合：

n        接口问题解决经验

n        数据抽取问题解决经验

n        数据转换问题解决经验

n        数据加载问题解决经验

数据质量知识的应用包括以下两个部分：

1.        在数据质量监控中的应用

检查规则是数据质量的标尺，通过对数据质量知识库结构化知识的分析，同时配合相应的采集和检查结果信息，系统可以自动产生检查规则的优化建议，优化建议经人工审计后，自动更新经营分析系统中对应的检查规则。

如通过分析三年中的某月指标波动情况，发现该指标环比波动率的上下限为5%和-3%，且这种波动是正常可接受的，而当前该检查规则的波动率阈值上下限为：3%和-2%，那么就可以根据实际情况调整该检查规则的波动率阈值上下限为：5%和-3%，以避免产生不必要的告警。

2.        在数据质量问题处理流程中的应用

n        在问题生成之前，可以自动或通过人工辅助方法在数据质量知识中检索是否有相同或相似问题，预先生成供参考的解决方案；

n        在问题分析环节中，可以检索数据质量知识，查看是否存在相同或相似的问题，以便于对问题的分析与定位；

n        在问题处理环节中，可以利用数据质量知识库中相同或相似的问题及其解决方案，及时有效地采取处理措施解决问题。在问题处理完后可提交该问题或该类问题的解决方案，以供后续知识的生成；

n        在问题总结环节中，采用归纳、演绎和推理等方法把问题与解决方案组成知识内容。在对问题总结时可以在数据质量知识库中检查是否有重复问题的发生，避免产生重复的知识。

3.2.9.3   数据质量知识内容

每条数据质量知识可以由以下三个部分的内容组成：

1.      检查规则：检查规则是发现数据质量问题一个标尺，因此可以成为数据质量知识的一部分。检查规则包括的内容项有：检查规则名称、检查规则类型和检查规则内容（如算法、参数等）等。

2.      数据质量问题：通过数据质量监控、数据质量评估或人工发现的数据质量问题是数据质量知识的重要组成部分。数据质量问题包含的内容项有：问题产生来源（数据质量监控、数据质量评估和业务人员发现的问题）、问题描述、问题级别、问题所关联的对象、问题发生时间和问题负责人等。

3.      解决方案：解决方案是数据质量知识的核心，解决方案是针对数据质量问题所提出的处理方法或原则，解决方案的组成内容可以是文本，也可以是图片、表格或文件附件。解决方案的内容项包括：解决方案内容和附件等。

4       数据质量管理流程

数据质量管理流程是与数据质量管理机制各个环节紧密相关的，是数据质量管理机制落地的载体。

数据质量管理流程贯穿数据异常事件或数据质量问题从产生到处理、解决的全过程，各个控制环节包括事件源头产生、事件分类、事件分析和原因定位、审批、实施、及最终的结果验收都应设置相应的控制流程。

可遵循的数据质量管理流程包括：

n        需求开发质量管控流程

n        数据源变更流程

n        规则变更流程

n        问题处理流程（区分监控告警处理子流程，缺陷改进处理子流程）

n        信息发布流程

n        申告处理流程

n        数据质量考评流程（包括考核办法制定流程，考评结果发布流程）

n        知识管理流程

4.1     需求开发质量管控流程

需求开发质量管控是数据质量控制的重要部分。通过需求管理制度和流程的规范，实现需求分析、设计、开发、上线等方面与相关需求部门的协同质量管理，达到过程质量控制的目的。需求开发质量管控各子流程的关系总图如下：

 

需求管理流程中涉及需求提出方、需求审批人员、经分系统开发组、需求质量管控组。其中需求提出方包括市场、数据、集客、网络、客服、计费、财务等相关部门。需求审批人员包括需求提出人对应的部门领导、和经分系统项目负责人。

4.1.1    需求提交与受理流程

当业务部门或其他相关部门人员因工作所需、提出经分系统需求时，需求提出人需在经分系统发起需求提交流程（根据各省公司的实际情况，需求管理流程也可能统一在BOMC或专用的需求管理平台实现，此处以经营分析系统为例，不作硬性要求。下同。），经相关的部门领导和经分系统项目负责人审批后方能进入需求开发流程，一旦进入需求开发流程、则需求内容将不能进行修改。需求提交流程中可以根据需求重要程度和影响范围选择部门领导进行审批。

n        需求提出人的职责：

 需求提出人必须通过经分系统的需求提交流程提交，需求内容需要明确：需求目的、需求的业务类型、应用生命周期的使用期限、使用人员权限（人员的部门、姓名、工号）、应用展现位置、涉及的指标口径等。

n        领导小组的职责：

领导小组在该流程中主要承担审批人员的角色，一般由需求提出人对应的部门领导和经分系统项目负责人担任。

1）需求提出人对应的部门领导：应对本部门提交的需求内容给予审核，对不符合要求的需求退回需求提出方，审批通过的需求提交系统项目负责人进行审批。

2）经分系统项目负责人：应对接收到的需求进行审核，对不符合项目开发要求的需求退回需求提出方，审批通过的需求转入需求开发流程。

4.1.2    需求开发与管控流程

需求提交与受理流程结束后将进入需求开发与管理流程，经分系统项目负责人将需求分发至经分系统开发组和需求质量管控组。经分系统开发组需在规定时间内完成需求分析、设计、开发工作；需求质量管控组人员在各个步骤设置控制点进行需求开发质量监督。

n        经分系统开发组的职责

经分系统开发组收到符合要求的需求后应遵照需求中的要求,依照经分系统开发规范、及应用生命周期管理规范进行开发支撑。应按照与需求提出部门协商后的时间要求完成，确保经分需求和应用的名称、编码、属性等信息完整记录,并对需求中数据统计的合理性和准确性负责。

n        需求质量管控组职责

需求质量管控组一般由需求提出人、经分系统项目需求负责人、经分项目开发负责人主要担任，由业务部门数据质量管理员、经分系统数据质量管理员、数据协同处理组监督和协助。

需求质量管控组在需求分析时承担抽查需求内容和指标定义的合法性与合理性，需检查开发过程是否与需求分析结果保持一致、是否有偏差或有歧义。

4.1.3    需求上线流程

需求开发完成后经测试、并由经分系统开发组各成员对需求结果进行审核，确认通过后可通过需求上线流程给予上线，即转入需求上线流程。需求上线流程包括上线部署、上线检查、上线结果审核和满意度反馈等。

n        经分系统开发组的职责：

经分系统开发组在需求上线过程中，主要承担需求上线部署申请、开发结果确认、上线部署和上线检查等职责。上线部署申请和上线检查一般由需求负责人承担，开发结果确认需由该需求的开发组相关人员承担，上线部署一般由经分系统运维支撑人员处理。

n        需求提出人的职责：

确认需求开发和上线结果是否与需求期望一致，并进行需求满意度评价。

n        需求质量管控组的职责：

需在上线检查步骤中抽查与确认上线内容的合法性。

n        业务部门数据质量管理员的职责：

监管需求质量管控组的工作，不定时地抽查需求质量管控组的工作情况，抽查部分上线需求内容是否合法。

4.2     数据源变更流程

通过数据源变更流程，能够让数据源系统与经分系统有效的进行协同处理问题。通过流程的支撑明确在数据源管理中各自的职责。

整个数据源变更流程包括如下六个部分：

n        变更提出：变更接入主要由三个部门提出变更，一个为业务部门，一个为数据源系统提出的变更，另外一个为经分内部自身变更；

n        变更影响分析：经分数据协同处理小组根据变更的提出，经分内部各系统（数据仓库、数据挖掘、数据集市等）负责人进行相应的影响分析，看看是否涉及到变更；

n        变更方案制定：经分协同处理小组负责人根据影响分析的结果，制定相应的变更方案，将方案评审，然后交由数据质量管理员进行审批；

n        变更实施：经分内部各系统根据数据源变更的方案实施数据源变更；

n        变更结果确认：需求提出人收到需求完成确认之后，进行变更确认。

n        变更结果归档：确保经确认的结果形成相应文档或记录，留底备案备查。

4.3     规则变更流程

通过数据质量监控规则变更流程，让数据质量监控过程中能够进行有序规则变更，通过规则的优化保证数据质量监控的准确性。

规则变更流程主要包含规则变更提出、规则变更影响分析、规则变更实施、规则变更反馈四部分。

 

n        规则变更提出：由数据质量监控组或者数据协同处理小组发现问题，并要求进行对监控规则进行变更处理。

n        规则变更影响分析：由数据质量监控人员与数据协同处理人员对规则的变更进行影响分析。

n        规则变更实施：由数据质量监控人员根据规则变更要求实施对应的规则变更，在规则实施过程中要求监控人员必须在24小时内完成规则变更的实施。

n        规则变更反馈：数据质量监控人员根据监控结果，对已经实施的规则进行反馈，通过意见反馈结果，进行审批，确认是否进行重新修订规则，如果不需要重新变更规则，则当前规则变更完成。

4.4     问题处理流程

数据质量问题根据发现途径进行分类，主要包括两大类问题：

1.        系统监控发现的数据问题；

2.        经分系统项目内部人员主动发现的数据质量问题。

根据数据质量问题的原因定位不同，触发不同的处理流程，由对应的执行小组进行解决。

n        接收数据问题处理请求：监控发现的问题由系统自动登记，自动生成故障处理任务；主动发现的问题由发现人登记问题情况，启动处理流程。

n        问题分析和解决方案评审：

1.        经分项目负责人：审批问题是否可改进和修复。

2.        质量问题处理组：对登记的问题进行分析，并提供相关解决方案。

3.        数据质量管理员：审批解决方案是否正确。对于系统故障等简单的问题，可以选择跳过解决方案评审环节。

n        问题解决处理环节：

1.        模型/代码问题：经分系统开发组进行设计、程序开发、测试，及完成配置项确认。

2.        数据源问题：通过数据源变更流程，由数据协同处理组解决。

3.        数据重传/重跑、数据处理、系统故障等其他问题：由质量问题处理组进行数据问题的修复处理，最后完成配置项确认。

n        问题修复完成确认：问题修复上线之后，由数据质量管理员对数据进行检查，最后确认问题是否已修复，并封闭流程。如果是主动发现的数据问题，数据质量管理员需与问题登记人确认问题是否已按预期目标进行修复。

4.5     信息发布流程

通过信息发布流程，实现数据质量的信息发布。

信息发布流程作为公共模块，多个流程可以共用该流程。

信息发布流程包括信息提交、发布审批、信息发布三个步骤。

 

n        信息提交：信息发布流程可以通过数据源变更、需求上线等流程进行触发，也可以通过质量监控人员手动触发，提交相应的待发布的信息。

n        发布审批：根据信息发布的要求，不同的信息发布内容需要不同的管理人员进行审批，如涉及到省KPI则由省公司数据质量管理人员审批，如果是市公司的KPI则由市公司数据质量管理员进行审批。

n        信息发布：经分支撑平台对发送的内容进行发送，发送的方式主要包括短信、邮件、系统公告等。

4.6     申告处理流程

申告处理流程，是由申告方在经分系统上提起数据申告，数据质量申告处理组根据申告内容进行问题分析、跟进申告处理至最终回复的流程。

 

1.        申告处理流程的登记与接收环节：

由申告方在经分系统上登记问题，经分系统记录后，由申告处理组接收申告、并进行初步分析，确认是否需要转交质量问题处理组。

n        经分系统：提供申告登记功能

n        申告处理组：在经分系统上接收申告，与客户沟通，并进行问题的初步分析，确定是否需要转交质量问题处理组核查。

2.        申告处理环节：

n        申告处理组：对较简单的申告问题进行分析，判断问题类型。对于数据源问题，则转数据协同处理组进行处理；对于口径咨询类问题，则直接回复；对于接口变更及代码开发问题，则转缺陷改进流程处理。对二次申告需重新处理。在整个申告处理过程中处于主导角色，并负责一直跟进处理和监督进度、直至回复。

n        质量问题处理组：接收申告处理组转来的质量问题核查任务并核查；牵头处理因接口变更或代码程序引起的问题和执行相关流程。

n        数据协同处理组：接收申告处理组转来的数据源数据质量问题，协调解决相关问题。

3.        申告回复（结束）环节：

n        申告处理组：跟进反馈提出的申告，跟进申告方的满意度反馈；

n        申告方：一般为业务部门、地市公司等使用经营分析系统人员。

4.7     数据质量考评流程

数据质量考评流程包括两部分，一部分为考核办法制定流程，实现考核标准的制定和发布；另外一部分为考评结果发布流程，定期通过系统发布或者手工发布的方式进行考评结果的发布。

4.7.1    考核办法制定流程

考核办法制定流程主要完成对数据源系统和经分系统开发团队考核规范的制定和发布，根据实际执行效果进行动态修订和完善。

被考核方：被考核方包括数据源系统和经分系统开发团队。根据数据质量管理方的要求，拟订适合帮助本系统或者本团队提升数据质量的考核规范，规范经数据质量管理方审批确认之后发布执行。

数据质量管理方：按照被考核方拟订的考核规范，根据经分系统数据质量管理要求进行审核和确认。

4.7.2    考评结果发布流程

n        系统支撑平台：提供考核需要的数据采集功能支撑，可以通过系统自动采集数据，也可以通过人工录入系统的方式。另外对于已经通过的考评数据进行数据的发布，并通知到被考评方。

n        数据质量管理方：对支撑平台提出的考核明细数据进行审批。对于被考核方提出的申诉进行核查，并进行重新审批操作。

n        被考核方：对系统发布的数据进行确认，如果不同意支撑平台提供的数据，可以进行相应的申诉，提供相应的申诉材料。

4.8     知识管理流程

知识管理流程用于规范数据质量管控工作中的知识归档和经验积累，为后续的数据质量管理工作提供有效的实践案例和专家意见。

n        数据质量监控人员、申告处理人员：通过数据质量监控、数据质量申告，数据质量评估、业务或者维护人员手工提交问题等途径产生数据质量问题，在问题生成过程中，无论是自动生成还是手工提交的问题，都可以先检查数据质量知识库中是否有相同或类似的问题，生成建议的解决方案。

n        数据质量处理人员、数据源管理人员：在对数据质量问题进行处理的过程中，可以采用问题生成或分析阶段产生的解决方案，或重新在数据质量知识库中寻找相同或相似问题的处理方案，也可以在已有的解决方案基础上进行修正或产生新的解决方案。根据解决方案，采取问题处理的措施，使经营分析系统和相关数据回归正常状态。当问题处理环节结束后，需要对问题处理过程进行总结，对问题处理的结果进行评估，对共性的、普遍的问题及其解决方案进行归纳，形成问题处理案例知识。

n        知识库、专家组：应用经分知识库的生命周期更新、知识协助编辑、知识评价等功能对知识进行审计，将数据质量问题处理过程中产生的知识进行分类、评估、归纳，形成新知识或者完善已有知识，用于后续质量问题的参照和质量管理工作的改进。

n        知识应用：在数据质量问题处理的各个环节都可以使用已有的数据质量知识进行协助，产生数据质量问题建议解决方案。

5       省公司对总部经分的数据保障

中国移动经营分析系统采用两级系统架构，由位于总部的一级经营分析系统（以下简称“一经”）和位于省公司的省级经营分析系统（以下简称“省经”）共同组成。省经部署面向省公司的决策支持、市场监控、营销支撑、员工信息支撑和数据质量等相关应用，一经部署面向全网的应用。

省经依据规范要求向一经传送本省明细及汇总数据、元数据，辅助一经进行数据质量检查和问题确认，为一经提供数据质量保障。一经对省经上传的数据进行文件级检查、记录级检查和校验规则检查，以保证其数据质量。

5.1     接口规范开发管理

省公司需要建立一经接口规范开发管理流程来保障一经接口开发质量。

省公司一经接口开发管理员通过中国移动通信有限公司业务支撑系统部交流网站接收一经规范变更内容，并根据总部上线要求制定开发计划。

需求管理阶段，省公司一经接口开发管理员、一经接口数据质量管理员、源系统数据质量管理员、业务部门数据质量管理员一起参与接口需求评审，结合规范要求和省内业务开展情况确定省内技术和业务口径，并定义数据质量稽核规则，纳入需求开发和省数据质量管理系统中监控。

需求开发完成后，由省公司一经接口开发管理员、一经接口数据质量管理员、源系统数据质量管理员、业务部门数据质量管理员进行联合测试，并和省经指标、财务定报指标等进行差异比较，保障数据一致性。

上线试运行阶段，由省公司一经接口开发管理员牵头和数据质量管理员共同负责数据质量管理，并解决试运行过程中发生的问题。试运行阶段完成后由省公司一经接口数据质量管理员负责接口日常运维工作。

5.2     日常维护管理

省公司需要建立一套完备的管理办法保障上传一经接口数据的数据质量。

5.2.1    接口质量保障

5.2.1.1   接口日常管理

省一经接口数据质量管理员负责对一经接口进行日常监控，通过省数据质量管理平台实现对一经上传及时性的监控和一经接口准确性的稽核（也包括来自总部反馈的数据质量问题）。一经日常监控不仅监控接口的最后上传时间和准确性稽核，同时监控从源系统到一经接口生成的各个环节，设立各自的完成时间要求和准确性稽核规则，准确性稽核规则包括总部一经的考核规则，同时包括省内业务的稽核规则、省经和一经接口数据和指标的一致性稽核规则等多种准确性规则。

省一经接口数据质量管理员负责对监控中发现的异常进行登记，并评估是否对一经接口数据造成影响，对可能造成影响的问题提交省数据质量管理小组，进入问题处理流程。省一经接口数据质量管理员根据省数据质量管理小组的处理结果评估对一经接口的影响，并形成解决方案，提交总部一经管理员进行审核。审核同意后由省数据质量管理小组牵头进行问题解决，省一经接口数据质量管理员跟踪验证问题的解决情况，并将解决结果提交总部一经管理员进行确认。

省公司在一经接口数据错误需要重传接口，或需要进行快照下发和数据发布时，由省一经接口数据质量管理员在一经数据质量管理平台的接口处理单模块提交申请，申请中需要明确处理类型、涉及的接口、数据周期，同时描述产生的原因。总部一经接口管理员对申请单进行审核，处理完毕后工单流转到省一经接口数据质量管理员处进行后续省内一经接口处理。

5.2.1.2   设备故障处理

省公司涉及一经设备发生故障后，省系统管理员需要在4小时内通知省一经接口数据质量管理员，并评估硬件影响和预计恢复时间。省一经接口数据质量管理员接到通知后评估对一经接口的影响，并通过电话和邮件方式通知总部一经管理员。如果故障影响到一经接口及时性或指标异动，需通过一经数据质量管理平台的预警上报功能在12小时内上报相关影响，并制定故障恢复后的数据恢复方案。故障恢复后省系统管理员需要在1小时内通知省一经接口数据质量管理员，省一经接口数据质量管理员通知总部一经管理员，并根据制定的数据恢复方案进行应用恢复。

总部一经系统出现故障后，通过一经数据质量管理平台发布受影响的范围和时间。省一经接口数据质量管理员接到通知后，根据总部统一安排制定省内一经接口上传保障方案。

5.2.2    质量问题自查

省一经接口数据质量管理员根据一经接口上传情况，进行省内自查，并填写质量问题自查报表。

（1）    日自查报表

根据每日一经接口质量监控报告，填写每日上传文件情况，对数据上传延迟和文件级、记录级报告错误填写问题产生原因和解决办法，需要变更优化内容提交省一经接口开发管理员进行开发；

根据每日关键指标波动性和一致性检查要求，对超过阀值的指标确定指标异动原因。省一经接口数据质量管理员需通过比对源系统、省经和一经数据一致性首先确认是否系统问题。系统方面问题需联合源系统和省经质量管理员确定原因和解决方案；非系统原因，需提交省数据质量管理小组分析数据质量原因并进行验证。

每日在规定时间前登录一经数据质量管理平台确认当日自查报表是否填报完毕。

（2）    月自查报表

根据月自查模版上报综合检查和数据业务指标内容。并设置数据波动阀值，省一经接口数据质量管理员需要对数据异常波动或违反稽核规则的内容提交省数据质量管理小组分析数据质量原因，并提交整改方案。

（3）    集团客户指标核查

省一经接口数据质量管理员根据集团客户指标异动情况联合集团客户部业务管理人员分析原因，省一经接口数据质量管理员分析是否系统问题，集团客户部业务管理人员分析是否业务开展问题，并形成原因分析结果和整改方案。

省一经接口数据质量管理员需登记所有的问题，并跟踪问题的解决情况和进行验证，促使问题最终解决。

5.2.3    质量问题回复

总部对发现的一经数据质量问题在一经数据质量管理平台的问题流转中心模块发布问题调查，并通过短信提醒方式通知省一经接口数据质量管理员。

省一经接口数据质量管理员在接收到问题后，对不清楚的内容或口径问题和总部问题发布人及时沟通，确定发布问题内容。并转入数据质量问题处理流程，同时跟踪处理的各个环节，在规定的时间内完成回复。

对问题回复被打回的情况，省一经接口数据质量管理员需要和总部问题发布人沟通，确认被打回的原因，并重新进行问题处理，在规定的时间内完成回复。

省一经接口数据质量管理员需要每日至少安排两次登录一经数据质量管理平台检查是否存在问题未回复情况。

5.2.4    预警上报

省一经接口数据质量管理员需要收集可能影响一经数据质量的各种因素，包括：源系统上线、系统割接、数据批量调整、市场营销活动、省指标业务口径和技术口径调整等，并通过数据质量管理小组评估是否对一经指标造成影响。

对可能造成影响的内容需提前5个工作日在一经数据质量管理平台预警上报模块进行上报，上报可能导致的影响范围和影响时间，并和总部一经管理员进行沟通，得到认可后再开展相应的调整。

5.3     源系统割接引发数据同步的操作指引

省公司在进行BOSS等支撑系统或支撑平台割接过程中，可能存在由于三户模型重新构建而导致用户ID、客户ID、渠道ID等基础标识的变更，需要由总部和省公司一起对一经数据仓库中相应的标识数据进行必要的互动操作，以保证一经和省经的同步数据更新，并保障各省当日及以后的各类应用指标正常。

5.3.1    割接影响评估和方案上报

省公司在系统割接前，需要提前对评估是否对一经系统产生的影响，涉及一经系统配合调整的必须提前5个工作日提交割接方案，在一经数据质量管理平台通过接口处理单上报。

5.3.2    变更数据上报

省公司在割接方案中需要确定受影响数据的范围和割接数据日期：

（1）    对于全省ID都受影响的情况，一经将统一全量修改；

（2）    对于部分ID受影响或分批割接的情况，省公司需要在割接前一日以文件接口方式提供一经需要修改的ID，文件名为“省份编码_接口编码_上传日期.txt”，文件中只包含一列数据，即将要割接的数据的旧ID，文件上传至省端接口机/tmp目录下。

5.3.3    日接口处理机制

对于日增量接口，一经系统在割接前一日接口数据处理完毕后，将需变更的数据置为失效，失效日期为省公司上传新数据的数据日期。省公司将新ID标识的有效数据以日增量方式上传，其中对于用户相关接口不要再上传历史离网用户，而是只传当前有效用户数据，保证相关日指标（如当日离网等）不会发生较大异动。

日全量接口正常不会受到影响，正常上传数据；

对于在实际割接中并不发生变化的接口或数据，省公司仍按照正常方式上传。

同时，为保证一经数据与省公司割接后数据的一致性，省公司应于当月内完成该部分数据的同步工作。在割接结束后向集团上传截至割接前最后一日的所有有效（在网或使用）和割接日前的当月失效（离网或停用）的新老ID对应关系文件（根据不同标识类型的分别提供），文件内容包含两个字段，分别为“割接前的旧标识”、“割接后的新标识”，文件上传至省端接口机/tmp目录下，上传后通知一经集成商（TERADATA和亚信）。如果省公司需要总部先下发截至割接前最后一日的所有在网和当月离网的旧标识列表，需提前5个工作日通过一经数据质量管理平台填写接口处理单进行申请，待总部下发数据后再传相应的新旧映射文件。

5.3.4    月接口处理机制

正常情况下对月接口不产生影响，按原有方式上传即可实现平滑过渡。一经根据接口数据变化方式进行相应处理。

5.3.5    1%抽样列表和拍照用户处理机制

省公司在割接前一日数据上传完毕后，更新集团于当月下发的1％抽样用户ID列表，将割接后的在网用户替换成新用户标识（对于割接前的历史离网用户仍保持旧用户标识）。完成列表更新后，按照新标识数据生成1％抽样话单接口文件。

一经在割接日更新上年年底拍照用户（中高端拍照用户、集团拍照客户等），将有效数据替换成新标识（对于割接前的历史离网仍保持旧标识）。完成列表更新后，按照新标识数据完成相关分析应用。

同时，为保证一经1％抽样用户数据与省公司割接后数据的一致性，省公司应于当月内完成该部分数据的同步工作。在割接结束后向集团上传割接后在网的1％抽样用户标识的新老对应关系文件，文件名为“省份编码_1PCT_MAP_上传日期.txt”，文件内容包含两个字段，分别为"割接前的旧用户标识"、"割接后的新用户标识"，字段格式为20位定长。文件上传至省端接口机/tmp目录下，上传后通知一经集成商（TERADATA和亚信）。

5.4     数据质量考核

5.4.1    考核管理办法综述

一级经营分析系统数据质量考核是中国移动业务支撑网业务运营质量考核管理体系的有机组成部分，目的是通过考核一经省级接口数据上传的及时和准确程度，以及相关数据质量管理工作的响应程度，反映业务支撑网对经营分析工作的支撑能力。

具体考核细则参见《中国移动业务支撑网业务运营质量考核管理办法》。

5.4.2    考核管理中的权利义务关系

有限公司负责全网经营分析系统数据质量的管理和业务指导，对考核管理办法负有颁布和解释权，对考核结果负有仲裁和决定权，并保留考核管理的最终否决权，在每年的业务支撑评奖中设立否决项，对有重大数据质量问题，或引起总部高层关注的问题省公司取消相关评奖资格。

省公司对上传总部一级经营分析系统的数据质量（即数据的及时性、完整性、一致性、准确性和真实性）负有直接的保障责任和义务。同时，省公司对考核管理办法的优化改进负有建议权，在考核管理执行过程中拥有对疑问事项申告的权利。

5.4.3    考核成绩发布与申告处理

每月中有限公司根据考核管理办法和校验规则出具上一月数据质量及保障管理工作响应的考核初评成绩，通过业务支撑交流网站进行明细预发布，并正式通知各省公司。

省公司根据预发布成绩，结合本省实际工作情况，综合分析初评结果的合理性，对存有疑义的项目向有限公司发起申告（需包含相关证明材料）。

有限公司考核仲裁委员会综合分析、评估省公司的申告材料，给出申告调整意见。有限公司考核经办人员根据上述调整意见进行考核成绩调整，并向部门提交最终考核成绩及调整说明。

考核结果经部门生产分析会审核通过后向省公司正式发布（通报公文）。
